JDK Toolchain Essentials

  • javac compiles .java source into portable .class bytecode
  • java starts the JVM, locates the entry-point class, and executes bytecode
  • jar packages classes/resources; javadoc generates API documentation
  • Since Java 9, distributable runtimes can be created with jlink using modules you require
  • Set CLASSPATH or module path so the JVM can find dependencies at runtime

See examples in ch01/examples

Compilation & Execution Flow

Compilation
  • Source → javac → bytecode
  • Compiler checks syntax, types, definite assignment
  • Errors must be resolved before running
Execution
  • JVM loads classes, verifies bytecode, and executes instructions
  • Uses class path or module path to locate referenced types
  • Runtime exceptions arise while executing (e.g., NullPointerException)

See examples in ch01/examples

Packages & Imports

  • Packages provide hierarchical namespaces (e.g., package ch01.examples;) and control accessibility
  • package declaration must match directory structure and precede other code
  • import statements allow simple type references; the default package should be avoided
  • Use wildcard imports for related utilities but prefer explicit imports for clarity
  • java.lang is imported automatically; all other packages require explicit import or FQCN

Class & Main Method Anatomy


package ch01.examples;

public class Zoo {
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}
  • Public class name must match file name (Zoo.java)
  • Entry point: public static void main(String[] args); args may be empty but must exist
  • Only one public top-level class per file; additional classes are package-private
  • Statements end with semicolons; braces define class/method blocks

Naming & Identifier Rules

  • Identifiers may use letters, digits, underscores, and currency symbols but cannot start with digits
  • Reserved words (class, static, this) and literals (true, null) cannot be identifiers
  • Naming conventions: UpperCamelCase for classes/interfaces, lowerCamelCase for methods/variables, UPPER_CASE for constants
  • Identifiers are case-sensitive; avoid similar-looking characters (e.g., O vs 0)
  • Use descriptive names to communicate intention

Primitive Data Types

Integral
  • byte (8-bit)
  • short (16-bit)
  • int (32-bit, default integer type)
  • long (64-bit, append L)
Floating & others
  • float (32-bit, append F)
  • double (64-bit, default decimal type)
  • char (16-bit Unicode)
  • boolean (true/false)
  • Numeric literals support underscores (1_000_000) and binary/hex/octal prefixes
  • Field primitives have default values; local variables must be initialised before use

See literals and assignments in Swan.java

Reference Types & Wrappers

  • Reference types point to objects on the heap (classes, arrays, interfaces, enums)
  • Wrapper classes (Integer, Double, etc.) provide object representations of primitives
  • Autoboxing/unboxing automatically converts between primitives and wrappers; beware of null
  • String is immutable; operations produce new instances rather than modifying existing ones
  • Use equals() for logical comparison; == compares references

Reference demos in Swan.java

Local Variables & Initialization

  • Local variables must be definitely assigned before first use; the compiler detects uninitialised accesses
  • Multiple declarations in a single statement must share the same type (int a = 1, b = 2;)
  • final locals can be assigned once; they are useful for constants or capturing values
  • Array declarations may specify brackets with the type (int[] values) or variable name (int values[])
  • Use descriptive initial values to clarify intent

See examples in NameTag.java

Type Inference with var

  • var reduces verbosity for local variables when the right-hand side clearly expresses the type
  • Cannot be used without an initializer or for fields, method parameters, lambda formals
  • The inferred type is fixed; subsequent assignments must be compatible
  • Use var when it improves readability, otherwise prefer explicit types
  • Be cautious with var when the inferred type is not obvious (e.g., var data = List.of(1, 2))

Inference samples in ZooVar.java

Scope, Blocks & Shadowing

  • Blocks ({ ... }) define visibility of local variables; once the block ends, variables go out of scope
  • Variables declared inside loops or conditional branches exist only within those constructs
  • Shadowing occurs when an inner scope declares a variable with the same name as an outer scope variable
  • Fields (instance/static) remain available for the life of their object/class regardless of method scope
  • Effectively final locals can be captured by lambdas and inner classes

See examples in ch01/examples

Object References & Garbage Collection

  • References can be reassigned, allowing multiple variables to point to the same object
  • Objects become eligible for garbage collection when no live references reach them
  • Setting references to null can release objects earlier, but usually the GC manages this automatically
  • System.gc() is only a hint; the JVM decides when to reclaim memory
  • Manage external resources (files, sockets) with try-with-resources or explicit close methods

GC examples in Swan.java

Key Takeaways

  • Master the toolchain to compile, package, and run Java applications effectively
  • Structure source files with correct packages/imports and consistent naming conventions
  • Choose between primitives and references appropriately; leverage wrapper classes when needed
  • Apply var judiciously and respect scope rules to avoid bugs
  • Understand how the JVM manages object lifetimes and garbage collection
