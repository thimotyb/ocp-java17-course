Lambda Syntax

  • Parameter types may be explicit, inferred, or declared with `var` (but be consistent across the list)
  • Single-expression bodies omit braces and `return`; block bodies must `return` explicitly when required
  • Use parentheses for zero or many parameters; a single inferred parameter may omit them
  • Match the target functional interface signature—parameter count and return type must align

Custom Functional Interfaces

  • A functional interface declares exactly one abstract method (SAM); annotate with `@FunctionalInterface` to detect mistakes
  • Default and static methods enrich the API without violating the SAM contract
  • Lambdas assign to functional interface references, enabling concise callbacks and strategy implementations
  • Functional interfaces can extend others as long as the SAM requirement holds

Built-in Functional Interfaces

Core interfaces
  • `Predicate` → boolean tests (`and`, `or`, `negate`)
  • `Function` and `UnaryOperator` for transformations
  • `Consumer` and `BiConsumer` for side-effect actions
Suppliers & primitives
  • `Supplier` defers value creation
  • Primitive specialisations (`IntPredicate`, `DoubleFunction`) avoid boxing
  • Compose functions via `andThen`, `compose`, and `identity()` helpers

Filtering & Composition

  • Combine predicates to express complex rules (`p1.and(p2).negate()`)
  • Map from domain objects to projections with `Function` and reuse the same pipeline across collectors
  • Leverage method references for readability when lambda bodies simply delegate
  • Reuse functional interfaces in streams (`filter`, `map`, `forEach`) and optional chains

Method References

  • `Type::staticMethod` for static calls, `object::instanceMethod` for bound receivers
  • `Type::instanceMethod` defers the receiver to the lambda parameter list
  • `Type::new` references constructors, useful for suppliers and factories
  • Use them whenever a lambda body simply delegates to an existing method for clarity

Lambda Scope & Effectively Final Variables

  • Lambdas capture local variables by value; captured variables must be effectively final
  • Reassigning captured locals is illegal—wrap mutable state in holders (`AtomicInteger`, arrays`) if needed
  • `this` and `super` inside a lambda refer to the enclosing instance, not to a synthetic lambda class
  • Avoid mutable shared state to keep lambda-based code thread-safe and deterministic

Key Takeaways

  • Match lambda signatures with functional interfaces and lean on `@FunctionalInterface` for validation
  • Prefer built-in `java.util.function` types for interoperability across APIs
  • Adopt method references when lambdas merely forward to existing methods
  • Keep captured variables effectively final to avoid compilation errors and concurrency bugs
